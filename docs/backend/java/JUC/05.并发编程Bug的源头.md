# 并发编程 Bug 的源头

## 一致性

在多 CPU 环境中，每个 CPU 都有自己独立的缓存，CPU 在执行过程中会将内存中的数据读取到自己独立的缓存中进行处理。

由于处理的结果暂存在自己独立的缓存中，其他 CPU 无法感知到数据被修改。

当变量 A 被线程 A、B 分别读取到自己的缓存中处理。A 处理完成后的结果对 B 线程来说是不可见的，这就造成了不同 CPU 之间的缓存一致性问题。

## 原子性

通常出现在单 CPU 环境中，编程语言中一行代码可能对应底层操作系统的多个操作，比如 `a+=1` 在实际执行过程中可能分为三步：

1. 读取变量 a
2. a + 1
3. 写入变量 a

这样一行代码就分成了三个部分，在其中任意一步执行结束后都有可能耗尽时间片导致线程上下文切换。假如线程 A 步骤 1 执行完毕后时间片耗尽，此时其他线程 B 也对变量 a 进行操作。线程结束后恢复到线程 A，此时线程 A 从步骤 2 继续执行，将 a+1 后写入磁盘。此时就导致线程 B 的操作结果被覆盖了。

## 有序性

我们编程的代码顺序并不一定是最终执行的顺序。例如 Java 创建一个变量 (New) 的正常步骤是

1. 分配一块内存 M
2. 在 M 上初始化对象
3. 将 M 的内存地址返回

但是编译器可能会做指令重排序，导致原有 1、2、3 的执行顺序变为：1、3、2。

双检查单例创建

```java
public class Singleton {
    private Singleton singleton;

    public Singleton getInstance () {
        if (singleton == null) {
            synchronized (Singleton.class) {
                if (singleton == null) {
                    singleton = new Singleton ();
                }
            }
        }
        return singleton;
    }
}
```

上述代码在 new 操作为 1、3、2 的执行顺序时，线程 A 在执行完 1、3 后，进行了上下文切换。此时线程 B 调用 `getInstance` 方法，因为 singleton 已经创建完毕所以直接返回了，但是对象并没有被初始化，所以会导致程序异常。
