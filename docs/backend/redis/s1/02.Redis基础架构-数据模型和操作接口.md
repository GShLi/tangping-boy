# Redis 基础架构

## 数据模型（能存储哪些数据）

1. 字符串
2. 列表
3. 哈希
4. 集合
5. 有序集合

## 基本操作接口（对数据可以有什么样的操作）

1. PUT：新写入或更新一个 Key-Value 对
2. GET：根据 Key 读取相应 Value 值
3. DELETE：根据 Key 删除整个 Key-Value 对
4. SCAN：根据一段 Key 的范围返回相应的 Value 值

在面对其他类型的 Value 时有更丰富的操作接口，例如 Redis 中的列表类型就有针对列表 Value 的操作。

## 基本模块

1. 访问框架
2. 索引模块
3. 操作模块
4. 存储模块

### 访问框架

Redis 的访问模式是什么？

常用的访问模式：

1. 通过函数库调用供外部应用使用（动态链接库）
2. 通过网络框架以 Socket 通信的形式提供键值对操作（Redis 采用的方式）

Redis 通过 Socket 对外提供服务，因此访问框架中包含 Socket Server 和请求解析两个部分。

### 索引模块

解析了客户端发来的请求后就知道了要进行的键值对操作，此时需要知道所要操作的键值对是否存在，这就依赖于索引模块。

索引的作用就是让键值数据库根据 Key 找到对应的 Value 存储位置，进而进行操作。

常见的索引类型：

1. 哈希表（Memcached、Redis）
2. B+树
3. 字典树
4. 跳跃表（RocksDB）

哈希表的优点：键值数据是存储在内存中的，内存的高性能随机访问可以很好的与哈希表的 O(1) 的复杂度相匹配。

### 操作模块

在找到对应的 Value 之后，如果 Value 是一个复杂类型，例如：列表、集合，还需要进一步的操作。

因此每种类型的 Value 的操作效率是不同的，取决于其实现结构。

### 存储模块

在找到 Key 对应 Value 的地址后，需要根据不同的操作执行具体的执行逻辑了。

GET、SCAN：直接返回 Value 值即可
PUT：如果是一个新值则需要开辟一块新内存空间并将值存储进去。
DELETE：需要将键值对删除并释放对应的内存空间。

上述操作中的 PUT 和 DELETE 就需要存储模块的支持了。

Redis 采用了不同的内存分配器。

持久化功能：

键值对通常存储在内存中，但是难免会遇到意外事件导致键值对数据库重启，如何能保证重启后能够快速的重新提供服务？

1. 每一个键值对都落盘保存
2. 周期性的将内存中的键值对落盘
