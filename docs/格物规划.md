## 二、未来文章分类规划

### 2.1 核心原则

1. **问题驱动，而非目录驱动**：不在没有实际问题时提前学技术
2. **实践层和思考层优先**：写决策过程、方案对比、失败复盘
3. **AI 时代的知识层内容不写**：概念定义、API 速查留给 AI

### 2.2 分类体系

```
series/
├── java/                          ← Java 生态
│   ├── language/                  ←   Java 语言
│   │   ├── juc/                   ←     并发 (13 篇)
│   │   ├── io/                    ←     IO/NIO (1 篇)
│   │   └── jvm/                   ←     JVM (4 篇)
│   ├── spring/                    ←   Spring（使用 + 源码，2 篇）
│   └── mybatis/                   ←   MyBatis（使用 + 源码，7 篇）
├── data-infra/                    ← 数据与中间件
│   ├── mysql/                     ←   MySQL (2 篇)
│   └── redis/                     ←   Redis (6 篇)
├── architecture/                  ← 架构设计
│   └── high-concurrency/          ←   高并发系统设计 (1 篇)
├── ai/                            ← AI 与开发 (占位)
├── engineering/                   ← 工程实践 (占位)

```

### 2.3 「AI 与开发」板块设计思路

这是专门给 AI 留的天地，定位**不是**：

- ❌ AI 概念科普（网上太多了）
- ❌ AI 算法原理（不是你的方向）
- ❌ 大模型训练/部署（另一个赛道）

而是：

- ✅ **AI 如何改变一个 Java 开发者的工作方式**
- ✅ **真实的使用体验和踩坑记录**
- ✅ **AI 辅助下的架构设计、代码审查、重构实践**
- ✅ **对 AI 时代程序员能力模型的思考**

#### 建议的前 5 篇文章

|------|---------|------|
| 1 | 《我如何用 Claude 辅助阅读 Spring 源码》 | 用 AI 解释复杂代码、生成调用链图、对比设计模式 |
| 2 | 《AI 辅助设计文档写作实践》 | 用 AI 帮你写 ADR，减少 boilerplate，聚焦决策本身 |
| 3 | 《当 AI 能写 CRUD，我们还剩什么》 | 对 AI 时代程序员价值的思考 |
| 4 | 《Prompt 即代码：一个开发者的提示工程笔记》 | 如何写好 prompt，如何把 prompt 纳入版本管理 |
| 5 | 《AI Review 代码的边界》 | 哪些问题 AI 能发现，哪些必须人工判断 |

## 三、内容优先级建议

| 优先级 | 板块 | 理由 |
|--------|------|------|
| 🔴 最优先 | AI 与开发 | 新兴领域，差异化强，传播力好 |
| 🔴 最优先 | 架构设计 (ADR) | 直接服务于职业路径 A（业务架构师） |
| 🟡 次优先 | Spring 手写系列重启 | 已有 2 篇，有动力继续，最能展示深度 |
| 🟡 次优先 | 读书笔记 | 输入倒逼输出，费曼学习法实践 |
| 🟢 日常积累 | 工程实践、调优案例 | 随做随写，不需要专门学 |

## 四、写作模板

每篇文章建议遵循以下结构（适配思考层和实践层）：

```markdown
---
date: YYYY-MM-DD
categories:
  - 格物
tags:
  - xxx
---

# 标题（体现"做了什么"或"怎么想的"）

## 背景
我遇到了什么问题 / 为什么要学这个？

## 过程
我是怎么做的，尝试了什么，哪些失败了？

## 方案
最终选择了什么方案，为什么选 A 不选 B？

## 反思
做完后有什么感悟？如果再遇到类似问题会怎么做？

## 参考
给后来者的资料链接
```
