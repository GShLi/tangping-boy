# 我用 Claude Code 重构了一个网站

## 背景

三年前我搭建了一个 VuePress 技术笔记网站，初衷是记录学习笔记。沿用一套仓库式管理慢慢内容变得臃肿，目录结构长这样：

```
series/00.basic/03.backend_language/01.java/03.java-juc/
```

光是目录名就有 5 层嵌套，再加上 193 个只有标题的空壳文章，构建一次要两分钟，每次打开都不知道自己在看什么。

但我一直拖着重构——建站一时爽，重构火葬场。

直到最近开始用 Claude Code。决定用它来"重构自己的网站"，当作 AI 辅助编程的实战。

## 第一步：把问题交给 AI 分析

把整个仓库丢给 Claude Code，问了一句：

> "分析一下我这个网站的内容，并给出建议。"

它花了大约 10 分钟扫描了 289 个 Markdown 文件，给出一份诊断：

> "该站点约有 30% 边界质量，25% 良好质量，其余 45% 是骨架/占位符。"

一条结论直接刺痛了我：

> "它相当于一个正在进行的、在其发展早期就被暂停的项目。"

说得太对了。

Claude 还发现了拼写错误（`architecture_hierachy` → `hierarchy`）、侧边栏标签暴露内部函数名（`genDataSource` 当作中文标签展示）、临时草稿文件混入仓库。

如果让人来做这些检查，至少半天起步。AI 做这个就像在数米粒，又快又准。

## 第二步：让 AI 出方案，不是出代码

我问 Claude Code："你觉得我该如何规划这个网站？"

它没有直接给代码，而是问了我一个问题：

> "你的兴趣爱好比较广泛——阅读、哲学、摄影，这些有没有必要加入到本网站中？"

这个问题改变了我对网站的定位。之前觉得技术博客就该只写技术。但 AI 说得对——AI 时代，一个"只会写 Java 的人"和一个"懂 Java、读过哲学、热爱摄影的人"，后者的不可替代性远高于前者。

我只做了一个核心决策：网站定位从"笔记仓库"转到"思考外脑"。AI 不差知识，但差判断和思考。

具体方案它是这么提的：

- 首页从空洞的"回顾过去 / 珍惜当下 / 展望未来"改为三个可点击内容入口
- 导航栏从「学习笔记」改为「格物」（出自《大学》"格物致知"）
- 学习笔记下分 7 大板块，为 AI 留一块天地
- 空壳文章要么删除，要么隐藏

我采纳了大部分。然后让它写了一份详细的执行计划，包括每个文件的改动列表。

## 第三步：执行——AI 写代码，人做判断

改造过程 AI 处理了绝大部分体力活：

- 批量重命名目录
- 为 17 篇博客文章添加 frontmatter（date、categories、tags）
- 重写导航栏配置
- 生成新的侧边栏

但调试阶段暴露了一些问题。

### 导航栏第 4 层

我设计了一个 4 层导航结构：

```
格物 → Java 生态 → Java 语言 → 并发 (JUC)
```

构建成功，访问却白屏，控制台报 `Missing required param "catchAll"`。

我本能地开始排查 base 路径、路由配置，折腾半天。最后问了 Claude Code 一句，它看了 vuepress-theme-reco 的 `DropdownLink.vue` 源码后回复：

> "导航栏模板硬编码最大 3 层。你的第 4 层 (grandchild) 渲染为 `<Link>` 组件，无法再嵌套。"

它建议将 JUC/IO/JVM 的合并放到侧边栏实现——导航栏只做入口，侧边栏做分组。

如果自己排查，我大概率要翻源码翻到天黑。AI 帮我省了这个时间。

### 博客页空白

重构后点开 `/posts.html` 发现主内容区空白。

这是我犯的一个错：把 `blogs/` 目录重命名成了 `posts/`。Claude Code 翻了 `vuepress-theme-reco` 的 `PageCreater.js` 源码，发现它硬编码了正则匹配 `/.+\/blogs\/.+\.md$/`：

```js
// node_modules/@vuepress-reco/vuepress-plugin-page/lib/node/PageCreater.js
const publishFlag = !(!/.+\/blogs\/[(.+)\/]?.+\.md$/.test(page.filePath || '') ||
```

改回 `blogs/`，问题解决。

### 翻页路径双写 base

博客翻页时 URL 变成了 `/tangping-boy/tangping-boy/posts/2.html`。

Claude Code 定位到 `Posts.vue` 第 47 行：

```js
const homeHref = withBase('/posts')
router.push(page > 1 ? `${homeHref}/${page}.html` : `${homeHref}.html`)
```

`withBase('/posts')` 返回 `/tangping-boy/posts`，但 Vue Router 已经自动处理 base，导致双写。这是上游主题的 bug（rc.16 版本），直接 patch 了 `node_modules` 解决。

> 现在最新的 rc.26 版本可能已修复此问题。

## 结果

| 指标 | 改造前 | 改造后 |
|------|--------|--------|
| 总页数 | 278 | 107 |
| 空壳文章 | 193 | 0 |
| 构建时间 | ~120s | ~55s |
| 导航最大深度 | 混乱 | 3 层 |
| 目录可读性 | `00.basic/03.backend_language/01.java/` | `java/language/juc/` |

## 反思

### AI 做对了什么

1. **大规模分析**——289 个文件的内容质量检查，人做需要半天，AI 几分钟
2. **源码定位**——翻 `node_modules` 找正则、找模板逻辑，速度快且准确
3. **体力活**——批量重命名、添加 frontmatter、更新配置文件，零出错
4. **方案建议**——"格物"的命名、"暗房"的备选名、7 大板块的规划，这些需要知识广度的建议 AI 做得很好

### 人必须做什么

1. **做决定**——AI 可以列 10 个方案，但它不知道哪个对你好。"删除所有空壳文章"和"保留空壳作为占位符"之间，你必须选
2. **提供上下文**——AI 不知道你喜欢哲学和摄影，除非你告诉它。你的个人偏好是 AI 盲区
3. **判断"对不对"**——AI 说"将 JUC/IO/JVM 合并到 Java 语言"，你觉得对就执行，不对就调整。最终判断力是你的
4. **写这篇文章**——AI 可以生成重构总结，但它不知道这次经历对你意味着什么

### 最重要的一个体会

跟 Claude Code 协作的正确姿势，不是"给我重构网站"，而是：

> "这是我的网站，这是我的想法，帮我分析一下哪里需要改，然后我们来讨论。"

把 AI 当同事而不是工具。它不会累、不会抱怨、翻源码速度比你快 10 倍，但它需要你告诉它：你是什么样的人、想做成什么样的事。
