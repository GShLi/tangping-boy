# StateFulSet - 拓扑状态

## 无状态应用和有状态应用

Stateful Application

1. 各个应用实例之间有依赖关系：例如 应用 B 需要等待应用 A 启动后启动，应用 A 和 B 之间就存在依赖关系，因此这两个应用就是有状态应用。
2. 应用与外部数据之间有依赖关系：例如数据库应用 A 的实例依赖在本地存储的数据，当该实例被删除重建时需要回到删除前的状态以维系与外部数据之间的关联。

Stateless Application：

1. 各个应用实例之间没有依赖关系。
2. 应用与外部数据之间没有依赖关系。

## StatefulSet

Deployment 解决了什么问题，还有什么问题没有解决？

解决了一部分的应用编排问题。

为什么说是一部分呢？

Deployment 做了一个简单化的假设：Deployment 中所有的 Pod 都是完全一样的，他们相互之间没有顺序，也无所谓运行在哪台节点上。

需要的时候 Deployment 可以根据模板创建 Pod，不需要的时候也可以随时删除 Pod。

总的来说，Deployment 中 Pod 的启动顺序和部署位置是相对随机的 (受调度策略的影响)。

这种简单化的假设比较适合 **无状态** 应用。

为了支持有状态应用的编排，K8S 推出了 `StatefulSet`，将现实世界中的有状态应用抽象出了两种情况：

1. 拓扑状态：这些应用实例必须按照某种顺序启动，无论实例被删除重建多少次，启动的顺序都是一样的。
2. 存储状态：应用实例绑定的数据是固定的。无论实例被如何重建，其绑定的数据始终是同一份。


总结：

StatefulSet 的核心作用就是通过某种方式记录 Pod 的状态，在 Pod 被重建时能够恢复这些状态。
