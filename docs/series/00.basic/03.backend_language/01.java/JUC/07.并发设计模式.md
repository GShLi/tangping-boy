## Immutability 模式

::: tips
Immutability /ɪˌmjuːtəˈbɪləti/ n.不变；永恒性；不变性
:::

多个线程同时读写同一个共享变量会出现并发问题，这里的必要条件是读和写，若只有读没有写则不会存在并发问题。

所谓 Immutability 即让一个变量在创建之后永远处于只读状态。

Java 中可以利用 final 关键字标识类、方法、变量不可修改。

Java 中的 String、Integer、Double 等基础类型的包装类型都利用了 Immutability 来实现了线程安全。

如何提供修改功能？提供一个新的不可变对象。

### 享元模式

不可变对象的修改是创建一个新的对象，可能会造成创建的对象过多从而浪费内存，可以利用享元模式来解决这个问题。

享元模式可以将 `享元部分` 抽离出来复用。Integer、Long 等基础类型的包装类型内部都利用了享元模式。

例如 Long 对象的内部维护了一个静态对象池管理了 `[-128,127]` 之间的数字。

这也是为什么包装类型不能作为锁对象的原因，因为获取的可能是相同的对象。

```java
// 这里虽然创建了两个对象 a 和 b，但是这两个对象其实是相同的。
Long a = Long.valueOf(1);

Long b = Long.valueOf(1);
```

## Copy-On-Write 模式

观察 String 类的 replace 方法可以发现 replace 时并没有在原来数据上做修改，而是创建了一个新数组 buf，最后创建了一个新 String 对象。

这是因为存储数据的数组 value `private final char value[];` ` 是不可变的（Immutability）。

而 replace 采用的方式是将原有数据拷贝到新数组中再创建新 String 对象返回，这本质上就是 Copy-On-Write 模式。

而 Immutability 对象的修改操作往往都是采用 Copy On Write 模式实现的。

```java
public String replace(char oldChar, char newChar) {
    if (oldChar != newChar) {
        int len = value.length;
        int i = -1;
        char[] val = value; /* avoid getfield opcode */

        while (++i < len) {
            if (val[i] == oldChar) {
                break;
            }
        }
        if (i < len) {
            char buf[] = new char[len];
            for (int j = 0; j < i; j++) {
                buf[j] = val[j];
            }
            while (i < len) {
                char c = val[i];
                buf[i] = (c == oldChar) ? newChar : c;
                i++;
            }
            return new String(buf, true);
        }
    }
    return this;
}
```

在并发场景下，如果对读取的性能要求很高，写入的要求不高的情况下可以使用 Copy On Write 模式。

问题：Java 提供了 CopyOnWriteArrayList，为什么没有提供 CopyOnWriteLinkedList 呢？

1. CopyOnWrite 一般用在读取性能要求很高的场景，而 LinkedList 底层实现是链表，读取性能不高。
2. LinkedList 节点由于要维护前驱和后继节点对象的引用，相同节点数量下 copy on write 时会占用比ArrayList更多的内存。
