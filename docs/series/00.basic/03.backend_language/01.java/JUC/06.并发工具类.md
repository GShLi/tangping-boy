## Executor 与线程池

### 线程池工作原理：生产-消费

一般的池化资源的使用方式通常是调用一个类似 get 操作的方法获取资源，资源使用完成后再调用类似 release 的方法释放资源，使其重新回到资源池中。

Java Thread 执行任务的方式只有 run 方法，而 run 方法更改的方式只有**重写**或在构建 Thread 时**通过构造函数传递 Runnable** 两种方式，因此线程池不能使用一般池化资源的设计方案。

```java
class MyThreadPool {
    BlockingQueue<Runnable> workQueue;

    List<WorkerThread> threads;

    MyThreadPool(int poolSize, BlockingQueue<Runnable> workQueue) {
        this.workQueue = workQueue;
        for(int i = 0;i < poolSize; i++) {
            WorkerThread work = new WorkerThread();
            work.start();
            threads.add(work);
        }
    }

    void execute(Runnable command) {
        workQueue.put(command);
    }

    class WorkerThread extend Thread {
        public void run() {
            while(true) {
                Runnable task = workQueue.take();
                task.run();
            }
        }
    }
}
```

上方的代码简单实现了一个线程池，线程池中有一个阻塞队列用于实现生产者-消费者模式。

workQueue 用于存储等待运行的任务，我们向线程池提交任务，任务会被存储到 workQueue 中。

threads 存储正在运行中的特殊线程，线程的主要功能就是消费 workQueue 中的任务并执行。

### Java 中的线程池

ThreadPoolExecutor 则是 Java 并发包中提供的最核心的线程池工具。

其最完备的构造函数有 7 个参数：

```java
public ThreadPoolExecutor(
    int corePoolSize,
    int maximumPoolSize,
    long keepAliveTime,
    TimeUnit unit,
    BlockingQueue<Runnable> workQueue,
    ThreadFactory threadFactory,
    RejectedExecutionHandler handler) {
        // code
    }
```

**corePoolSize**: 线程池保有的最小线程数量。

**maximumPoolSize**: 当线程池中任务较多时，运行创建的最大线程数量。

**keepAliveTime & unit**: 线程池释放空闲线程界定的标准，当一个线程空闲了 keepAliveTime & unit 久且线程池中的线程数大于 corePoolSize，那么这个线程就要被回收了。

**workQueue**: 工作队列。

**threadFactory**: 自定义如何创建线程。ThreadFactory 是一个接口，包含 newThread 返回值是 Thread。

**handler**: 自定义任务拒绝策略。例如提交新任务时所有的线程都正在忙碌且有界工作队列也满了，那么线程池就会拒绝接收新任务。

ThreadPoolExecutor 拒绝服务策略：

* CallerRunsPolicy: 提交任务的线程自己去执行该任务。
* AbortPolity(Default): 会抛出 `RejectedExecutionException`
* DiscardPolicy: 直接丢弃任务，不抛任何异常。
* DiscardOldPolicy: 丢弃最老的任务，将最早进入工作队列的任务丢弃，将新任务加入到工作队列。

为何禁用： ***Executors*** ？

因为 Executor 提供的很多方法默认使用的都是无界的 LinkedBlockingQueue，在高负载的情况下容易导致 OOM。

## Future

在某些场景下我们需要知道任务的执行结果，Java 通过 ThreadPoolExecutor 提供了 3 个 submit() 方法和 FutureTask 工具类来支持获取任务执行结果的需求。

### ThreadPoolExecutor 的 submit 方法

```java
/**
 * 参数是 Callable 接口，他有一个 call() 方法，且方法有返回值。
 * 所以可以通过返回的 Future get 任务的执行结果。
 */
<T> Future<T> submit(Callable<T> task);

/**
 * 提交的是 Runnable，Runnable 接口的 run 方法是没有返回值的，
 * 因此这个接口返回的 Future 只能用于断言任务是否运行完成。
 */
Future<?> submit(Runnable task);

/**
 * 此方法与 Future<?> submit(Runnable task); 不同的是多了一个 result 参数。
 * result 相当于主线程和子线程之间的桥梁，通过 result 实现共享数据的功能。
 */
<T> Future<T> submit(Runnable task, T result);
```

Future 接口有 5 个方法：

```java
/**
 * 取消任务
 */
boolean cancel(boolean mayInterruptIfRunning);

/**
 * 判断任务是否已经取消
 */
boolean isCancelled();

/**
 * 判断任务是否已结束
 */
boolean isDone();

/**
 * 获取任务执行结果（阻塞）
 */
V get();

/**
 * 获取任务执行结果（阻塞），支持超时机制。
 */
V get(long timeout, TimeUnit unit);
```
