# 展望

> AI 时代，一个程序员的个人网站与职业规划。

---

## 一、网站定位：从「笔记仓库」到「思考外脑」

### 当前问题

网站现在主要是**知识搬运**——JUC 知识点、Spring 源码分析、K8S 概念等。AI 时代这些东西的获取成本趋近于零，问一句 Claude 就能拿到比你的笔记更完整的答案。

### 核心转变

**AI 能生成答案，但不能替你思考。** 网站应该记录的是：

| 不要写 | 要写 |
|--------|------|
| "Redis 有 5 种数据结构" | "我在 xx 场景用 String 做了缓存，遇到了热 key 问题，最终用 xx 方案解决" |
| "工厂模式的定义是..." | "读了《设计模式》后，我在重构 xx 模块时把 if-else 替换成了策略模式，改动前后对比" |
| "K8S Pod 是..." | 等你真正部署过再写 |

**一句话：写你踩过的坑、做过的决策、读过的书引发的思考。** 这些 AI 无法替你生成。

---

## 二、内容策略：三个层次

```
思考层  ← 决策过程、架构权衡、失败复盘（最有价值，AI 不可替代）
  │
实践层  ← 项目实战、问题排查、性能调优（有价值，展示解决能力）
  │
知识层  ← 概念笔记、API 速查、配置记录（价值最低，AI 可直接生成）
```

**建议删掉或归档 80% 的知识层内容**（空壳 K8S、纯标题设计模式等），把精力集中在实践层和思考层。哪怕只有 20 篇高质量文章，也比 300 篇骨架有价值得多。

## 三、内容架构规划

```
首页 (个人介绍 + 内容入口)
├── 📝 博客
│   ├── 技术分享    ← 运维实践、问题排查、项目复盘
│   ├── 个人随笔    ← 思考、感悟、观点
│   └── 读书笔记    ← 阅读后的理解和反思
│
├── 📚 学习笔记     ← 体系化知识库（Java / Spring / MyBatis / Redis）
│
└── 🗂️ 项目         ← 项目文档（iPostYou / doublemall / Titan）
```

### 现有内容迁移策略

| 现有内容 | 目标位置 | 原因 |
|---------|---------|------|
| `blogs/` 下运维实践、问题排查 | → 博客（技术分享） | 这些是独立文章，非体系化笔记 |
| `blogs/interview/` | → 博客或归档 | 面试记录是时间线内容 |
| `series/` 下 Java/Spring/MyBatis/Redis | → 保留在学习笔记 | 体系化知识，不适合拆散 |
| `project/` | → 保留在项目 | 项目文档有独立价值 |
| K8S/设计模式等空壳 | 暂不展示 | 有内容后再恢复 |

## 五、网站作为职业杠杆

### 5.1 对外：作品集效应

- 招聘方/技术 Leader 搜到网站时，看到的应该是**你能解决什么问题**，而不是**你学过什么**
- 每篇博客末尾加上求职标签（如果有需要）
- GitHub + 个人网站联动：在 GitHub 项目 README 中引用网站上的设计文档，网站上嵌入 GitHub 仓库卡片

### 5.2 对内：学习加速器

- **费曼学习法**：当试图把一件事讲清楚，会发现自己哪里没真懂
- **建立搜索索引**：遇到过的 bug、查过的资料，写下来就是给自己未来的搜索快照
- **积累可复用资产**：设计文档跳槽时可以脱敏后直接用

## 六、职业方向建议

### 技术栈现状

Java / Spring / MyBatis / Redis / Docker / MySQL / ETL

### 两条可行路径

**路径 A：深耕业务架构（推荐）**

- 现在已经写了 Spring 源码分析、高并发架构笔记、产品设计文档
- 方向：**从写代码的人变成设计系统的人**
- 重点能力：系统设计、技术选型、领域建模、分布式架构
- 网站上多写：架构决策记录 (ADR)、方案对比、复盘总结
- 不可替代性远高于"只会写 CRUD 的 Java 程序员"

**路径 B：数据工程**

- Titan ETL 项目、大数据目录占位表明对这个方向有兴趣
- 方向：数据平台开发、数据治理、实时数仓
- K8S/大数据笔记全是空壳——要么真正投入学习，要么果断放弃

**建议选 A**：Spring 源码笔记和 iPostYou 产品设计文档质量明显高于其他内容，已经在这个方向上投入了最多精力，也有最好的产出。

## 七、AI 时代的程序员生存策略

### 不要做的事

- ❌ 不要和 AI 比写代码速度——比不过
- ❌ 不要花大量时间背 API 和语法——AI 记得更准
- ❌ 不要学"全栈"为学而学——AI 让通才贬值，专才 + 系统思维升值

### 要做的事

- ✅ **培养 AI 无法替代的判断力**：什么场景用什么方案、为什么选 A 不选 B
- ✅ **学会用 AI 放大自己**：用 AI 辅助 code review、架构分析、文档编写
- ✅ **提升沟通和表达能力**：能把复杂技术讲清楚的人永远是稀缺的（网站就是训练场）
- ✅ **积累领域知识**：电商（doublemall）、邮政（iPostYou）——领域知识 + 技术能力 = 不可替代

## 八、30 天行动计划

| 周 | 行动 |
|----|------|
| 第 1 周 | 删掉所有空壳文章，配置博客功能，写一篇「关于我」 |
| 第 2 周 | 从工作/项目中找一个最近解决的实际问题，写一篇复盘文章 |
| 第 3 周 | 选一本最近读的技术书，写一篇读书笔记（不是摘录，是**自己的理解**） |
| 第 4 周 | 整理 LinkedIn / 掘金 / 知乎等对外渠道，把网站链接放上去 |

---

**核心就一句话**：AI 时代，网站不应该是一个"我知道什么"的目录，而应该是"我做过什么、怎么思考的"的证据。
